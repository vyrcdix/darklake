# pub-md — user's guide

A static site that publishes a curated set of markdown files defined in a manifest. Sources can live anywhere reachable by URL; pub-md fetches them on a schedule, detects changes, and republishes the site via GitHub Pages.

## How it works

1. `sources.yaml` lists every document to publish, with its source URL and destination path under `docs/`.
2. `scripts/fetch.py` reads the manifest, downloads each source, hashes the content, and writes any changed file into `docs/`.
3. `scripts/build_nav.py` regenerates the site nav from the manifest (grouped by `section`).
4. MkDocs (with the Material theme) renders `docs/` into a static site.
5. GitHub Actions runs fetch + build_nav + build + deploy on every push, on a 15-minute cron, and on manual trigger.

Once GitHub Pages is enabled, the site is served at `https://vyrcdix.github.io/darklake/`.

## File layout

```
sources.yaml              # the manifest
scripts/
  fetch.py                # URL fetcher (gdrive support comes in step 4)
  build_nav.py            # regenerates the nav block in mkdocs.yml
docs/                     # rendered input — hand-written + fetched
  index.md                # landing page (hand-written)
  <fetched paths>         # generated by fetch.py
.cache/hashes.json        # upstream content hashes; committed so CI sees real changes
mkdocs.yml                # theme + extensions + auto-generated nav (between marker comments)
.github/workflows/
  publish.yml             # cron + push + manual triggers
requirements.txt          # mkdocs + material + pyyaml
```

## Adding a source

Edit `sources.yaml`:

```yaml
sources:
  - title: "Onboarding"
    type: url
    url: https://raw.githubusercontent.com/your-org/repo/main/ONBOARDING.md
    path: engineering/onboarding.md
    section: Engineering
```

Fields:

- `title` — page title shown in the nav and as the HTML `<title>`.
- `type` — `url` for now. (`gdrive` is coming in step 4.)
- `url` — the raw markdown URL. For GitHub, use `raw.githubusercontent.com`, not the `github.com` blob view.
- `path` — destination under `docs/`. Must stay under `docs/`; path-traversal attempts are rejected.
- `section` — optional grouping; consumed by the nav builder in step 3.

Commit and push. The site rebuilds within ~15 minutes (cron) or immediately if you trigger the workflow manually.

## Removing or renaming a source

1. Delete (or update) the entry in `sources.yaml`.
2. Delete the corresponding file under `docs/` and its key from `.cache/hashes.json`.
3. Commit and push.

The fetcher does not currently auto-prune orphaned files — tracked under step 5 polish.

## Publishing triggers

The workflow runs on:

- **Push** to `main` or `master`.
- **Schedule** — every 15 minutes (`cron: "*/15 * * * *"`).
- **Manual** — Actions tab → "Publish site" → "Run workflow".

Each run does:

1. Fetch every source.
2. Regenerate the nav block in `mkdocs.yml` from `sources.yaml`.
3. If anything changed (fetched docs, cache, or nav), commit it back as `chore: update fetched docs [skip ci]`.
4. Run `mkdocs build --strict`.
5. Deploy the resulting `site/` to GitHub Pages.

## One-time GitHub setup

1. **Settings → Pages → Source** = "GitHub Actions".
2. **Settings → Actions → General → Workflow permissions** = "Read and write permissions" (so the cron can commit fetched changes back).
3. Trigger the workflow manually once to confirm the full pipeline works.

## Local development

This repo uses [`uv`](https://github.com/astral-sh/uv) for environment management.

```bash
# one-time setup
uv venv .venv --python 3.12
uv pip install --python .venv/bin/python -r requirements.txt

# pull the latest from every source
.venv/bin/python scripts/fetch.py

# regenerate the nav block from sources.yaml
.venv/bin/python scripts/build_nav.py

# preview the site at http://127.0.0.1:8000
.venv/bin/mkdocs serve

# strict build (what CI runs)
.venv/bin/mkdocs build --strict
```

## Troubleshooting

**Fetch fails with `HTTPError`.** The source URL is wrong or unreachable. Open it in a browser; for GitHub, use the raw URL.

**`No changes.` but I just updated the source.** The upstream hash matches what was last fetched. Confirm the source actually changed; if it did and you want to force a refresh, delete that key from `.cache/hashes.json`.

**Strict build fails.** Usually a broken link or unrenderable markdown. Run `mkdocs build --strict` locally to see the offending warning.

**Workflow can't push fetched changes.** Check that *Workflow permissions* in Actions settings is set to "Read and write".
